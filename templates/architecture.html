How ScriptScope Works

A multi-agent script analysis system built for Bullet AI's content intelligence platform. This page explains every architectural decision — what we built, why, and why not something else.

01 — THE PROBLEM

Why Not Just One Prompt?

The naive approach: send the script to GPT with "analyze this" and parse the output. Here's what happens:

AspectSingle PromptMulti-Agent (ours)
Summary length1 sentence or 8 — no controlExactly 3-4 sentences (enforced)
Emotional arc"The tone is tense then hopeful"Per-beat data with intensity values
Engagement scoreRandom number, no calibrationRubric-based, each level defined
Suggestions"Add more conflict" (generic)Targets weakest scores, cites lines
Self-correctionNoneDebate catches blind spots
Output validityHope JSON parses correctlyPydantic schema enforced by API
💡
Core insight: By splitting analysis across specialist agents, each with a focused prompt and domain rubrics, we get dramatically better output quality. The LLM can be an expert at one thing instead of mediocre at everything.
02 — ARCHITECTURE

The Pipeline

Script → Parser (regex, no LLM) │ ├──► Story Analyst ────┐ ├──► Emotion Analyst ──┤ (3 agents in parallel) ├──► Engagement Scorer ┤ │ │ └──► Script Doctor ────┘ (reads other 3 outputs) │ Debate (2 rounds) │ Rewrite 2 variants │ Re-score each variant │ Final Report

6 Phases

  • Phase 1 — Parse: Pure Python regex. No LLM call. Extracts title, scenes, characters, dialogue beats, stage directions. Instant, deterministic, free.
  • Phase 2 — Parallel Analysis: Story Analyst, Emotion Analyst, and Engagement Scorer run simultaneously via asyncio.gather. Each produces structured JSON validated by Pydantic.
  • Phase 3 — Informed Analysis: Script Doctor reads all three agents' findings, then targets the weakest engagement factors and emotional flat spots specifically.
  • Phase 4 — Debate: 4 cross-agent matchups over 2 rounds. Agents CHALLENGE or SUPPORT with evidence, then REVISE or HOLD. Self-correction mechanism.
  • Phase 5 — Rewrite: Top 2 suggestions are applied as full script rewrites. Each variant is a complete rewritten script.
  • Phase 6 — Re-score: Each variant is scored by the Engagement Scorer. Score deltas show whether the suggestion actually improved the script. This is the RL-style feedback loop.
03 — THE AGENTS

Four Specialists, Not One Generalist

Agent 01

📖 Story Analyst

Summary, plot type (constrained to 9 archetypes), narrative structure as arrow chain, character roles. Prompt enforces exactly 3-4 sentences.

Agent 02

🎭 Emotion Analyst

Maps each dialogue beat to an emotion from a fixed 15-emotion vocabulary with intensity 0.0-1.0. Produces visualizable data, not vague descriptions.

Agent 03

⚡ Engagement Scorer

6-factor rubric scoring: hook, conflict, tension, depth, cliffhanger, resolution. Each score has 5 defined levels. Must cite specific dialogue lines as evidence.

Agent 04

✍️ Script Doctor

Reads other agents' findings first. Targets lowest engagement scores and emotional flat spots. Also identifies peak cliffhanger moment.

🔑
Why the Doctor sees other agents' output: Without cross-agent context, suggestions are generic ("add more conflict"). With it, the Doctor says "Arjun's emotional flatness at beats 2, 4, 6 (from Emotion Analyst) combined with hook score of 7 (from Engagement Scorer) suggests the opening needs a physical reaction before dialogue."
04 — PROMPT ENGINEERING

The Intelligence Is in the Prompts

Three prompt design techniques that make this system work:

1. Rubric-Based Scoring

Instead of "rate the hook 1-10" (which produces random numbers), each score level has explicit criteria:

OPENING HOOK
9-10Immediate mystery or physical jolt. Viewer cannot scroll past.
7-8Strong curiosity that takes a moment to land. Good but not instant.
5-6Interesting but passive opening. No urgency.
3-4Generic setup. Could be any story.
1-2No reason to keep watching.

This produces consistent, explainable scores. The LLM reads "Why now?" and calibrates against the rubric: "question hook, creates curiosity but lacks physical jolt — that's a 7-8."

2. Constrained Emotion Vocabulary

Without constraints, the LLM invents arbitrary emotions ("melancholic anticipation", "bittersweet resolve"). We constrain to 15 specific emotions:

shock · anger · guilt · hope · sadness · tension · vulnerability · uncertainty · joy · relief · fear · nostalgia · revelation · defiance · resignation

This makes output structured, comparable across scripts, and directly visualizable as a chart.

3. Weighted Scoring Formula

For short-form content (reels, shorts), hook and cliffhanger matter most — they determine whether people start watching and share. The overall score is not an average:

overall = (hook × 1.5 + conflict + tension + depth + cliffhanger × 1.5 + resolution) / 7

This domain-specific weighting shows understanding of how short-form content engagement actually works.

05 — THE DEBATE

Agents Challenge Each Other

After all agents analyze independently, they cross-review in a structured 2-round debate:

Round 1 — Challenge or Support

4 matchups, each targeting a specific topic:

Challenger→TargetTopic
Engagement Scorer→Story AnalystOpening & hook assessment
Emotion Analyst→Script DoctorCharacter emotional depth
Story Analyst→Emotion AnalystArc classification & ending
Script Doctor→Engagement ScorerScore calibration

Round 2 — Revise or Hold

Challenged agents must respond with one of:

  • REVISE — accept the challenge and update their findings
  • HOLD — maintain their position with reasoning
🎯
Real example from testing: The Engagement Scorer challenged the Story Analyst: "The hook isn't just 'an unexpected message.' The 5-year temporal gap creates immediate curiosity debt." The Story Analyst revised its summary to lead with the time gap. This is self-correction that a single-prompt system can't do.
06 — VARIANT GENERATION

Suggest → Rewrite → Re-score (RL-Style Loop)

Most script analysis tools stop at "here are some suggestions." We go further:

Original script → Engagement Score: 7.8/10 │ ├── Suggestion 1: "Strengthen the hook" │ └── Rewrite → apply suggestion → full new script │ └── Re-score → 8.3/10 (+0.5) ✓ │ └── Suggestion 2: "Break Arjun's composure" └── Rewrite → apply suggestion → full new script └── Re-score → 7.6/10 (-0.2) ✗

This creates a feedback loop: the system doesn't just suggest improvements, it proves which ones work by generating the improved version and measuring the score change.

Users see:

  • The rewritten script (collapsible)
  • Factor-by-factor comparison (which scores went up/down)
  • Score delta (green for improvement, red for regression)
  • Summary of what changed and why it helped (or didn't)
🔬
Why "RL-style": In reinforcement learning, you take an action, observe the reward signal, and learn which actions improve outcomes. Here, the "action" is applying a suggestion, the "reward" is the engagement score delta, and the "learning" is showing the user which suggestions actually move the needle. It's not full RL, but it's the same feedback principle.
07 — STRUCTURED OUTPUT

Pydantic + OpenAI JSON Schema Mode

Every agent returns structured JSON validated against a Pydantic schema. This is enforced at the API level — the LLM cannot return invalid output.

# The schema class EngagementFactor(BaseModel): name: str # "hook", "conflict", etc. score: float # 0.0 - 10.0 (validated) reasoning: str # Why this score evidence_lines: list[str] # Specific lines cited # The API call response = client.responses.create( model="gpt-4.1-mini", text={"format": { "type": "json_schema", "schema": EngagementFactor.model_json_schema(), "strict": True, # ← guarantees valid output }} )

Why this matters: Most submissions parse LLM text with regex or json.loads() and hope it works. Structured output mode means the response is guaranteed to match the schema — every field present, every type correct, every constraint satisfied. Zero parsing errors in production.

08 — TECHNICAL DECISIONS

What We Chose and Why

Model

GPT-4.1-mini

Structured output mode, 1M context, ~$0.01/analysis. Could swap to Claude or Gemini with a one-line change.

Framework

FastAPI + Jinja2

Pure Python. No separate frontend. Templates serve the UI directly. One process, one deploy.

Parallelism

asyncio.gather

3 agents run simultaneously. Reduces wall time by ~60% vs sequential. No Celery, no Redis, no job queue.

Parser

Pure Regex

No LLM for parsing. Deterministic, instant, free. Handles Title:, Scene, Dialogue, [stage directions] formats.

Schemas

Pydantic v2

Type-safe models for all outputs. Generates JSON Schema for OpenAI strict mode. Validates every response.

Deploy

Render.com (free tier)

One-click from GitHub. No Docker needed. Auto-deploys on push. $0/month.

What We Deliberately Didn't Build

PatternWhy Not
Orchestrator classFixed 4-agent pipeline. Nothing to orchestrate — it's 3 lines of asyncio.gather.
Planner / task decompositionEvery script gets the same analysis. No dynamic routing needed.
Intent classifierOne input (script), one output (analysis). One intent.
ReAct loopsEach agent does one focused job. No multi-step tool reasoning needed.
DatabaseStateless analysis. Results are returned immediately. History can be added with localStorage or Vercel KV later.
Agent base class hierarchy4 functions are cleaner than 4 classes inheriting from an abstract base. Less code, easier to read.
🧠
Design philosophy: Knowing when NOT to use an architecture pattern is more important than knowing how to use it. Every component earns its place. No orchestrator, no planner, no intent classifier — because the problem doesn't need them.

11 — LIMITATIONS & FUTURE WORK

What We'd Build Next

Current Limitations

  • English only — emotion vocabulary and rubrics are English-specific
  • Single-scene scripts — no multi-act screenplay support
  • Rubrics are heuristic — not trained on real engagement data (views, completion rates)
  • Fixed 2 debate rounds — not dynamic based on disagreement level
  • Latency ~60-90s — serial debate + rewrite calls
  • No persistence — results aren't saved between sessions

With More Time

  • Fine-tuned scoring model — train on actual engagement data from Bullet's platform
  • Multi-scene support — per-scene analysis + whole-script aggregation
  • Streaming responses — show each agent's output as it completes
  • Dynamic debate — more rounds when agents strongly disagree
  • Comparative analysis — analyze and rank multiple scripts against each other
  • Agent memory — spot patterns across scripts by the same author
